The “real” config lives in someone’s head and a wiki, never fully written down or reproducible.

The ceiling

Where Compose stops

Everything in that YAML runs on a single host. That assumption breaks the moment you go to production.

💉

The host dies

The machine reboots or fails. Every container goes with it. Who notices, and who restarts them somewhere else?

📈

Traffic spikes

You need ten copies of the API, spread across machines. Compose can scale on one box — then you run out of box.

🔁

Zero-downtime release

Ship a new version without dropping requests, and roll back instantly if it misbehaves. Compose just restarts.

🌐

Many machines

Ten servers, hundreds of containers. Who decides what runs where, and reshuffles when a node dies?

?
The missing role — something has to watch the whole fleet, place containers, restart them, scale them and heal them. That something is an orchestrator.

Worker node · up close

Three jobs on every worker

The node’s agent. Takes Pod specs from the api-server, tells the runtime to start containers, and reports status back. If kubelet is down, the node is NotReady.

Putting it together

What happens when you kubectl apply

Follow one Pod from your terminal to a running container — and why it stays running.

  • 1
    kubectl sends your YAML to the api-server, which validates it and writes the desired state to etcd.
  • 2
    The scheduler sees a Pod with no node and assigns it to the best-fit worker.
  • 3
    That node’s kubelet notices the assignment and tells the container runtime to pull the image and start containers.
  • 4
    kube-proxy wires networking so the Pod is reachable through its Service.
  • 5
    kubelet reports health back to the api-server; you see it as Running.
  • 6
    Controllers keep watching — if the Pod or node dies, the loop recreates the work. Forever.

The object model

Everything is an object

Pods, Services, Secrets, nodes — every single one is the same five-field envelope. spec is your wish; status is the cluster’s report.

any-object.yaml
apiVersion: apps/v1        # which API group + version
kind: Deployment          # what type of object
metadata:                 # who am I
  name: vote
  namespace: vote
  labels:
    app: vote
spec:                     # what I WANT (you write this)
  replicas: 3
status:                   # what IS (the cluster writes this)
  readyReplicas: 3
  • 1
    apiVersion — which API group and version. v1 for core objects (Pod, Service), apps/v1 for workloads.
  • 2
    kind — the object type. Together with apiVersion it tells the api-server which schema to validate against.
  • 3
    metadata — name, namespace, labels, annotations. Identity.
  • 4
    spec — desired state. This is the only part you write.
  • 5
    status — observed state. A controller writes it; you only ever read it.
?
Why it matters — Because this shape never changes. Learn five fields once and every object you meet for the rest of the course — ConfigMap, PVC, HPA, NetworkPolicy — is already familiar.

Resource hierarchy

How objects relate — the hierarchy

Objects are not a flat pile. Most are owned by another object, and each one is either cluster-wide or scoped to a namespace.

Deployment
you create this
→

→

Ownership — who makes whom
  • Each object records its parent in metadata.ownerReferences.
  • Delete the parent and garbage collection takes the children with it — that is why deleting a Deployment removes its Pods.
  • Other chains: Service → Endpoints, PVC → PV, CronJob → Job → Pod, HPA → Deployment.
  • You almost never create the middle of a chain by hand.
Two scopes — where an object lives
  • Namespaced — lives inside one namespace. Pod, Service, Deployment, ConfigMap, Secret, PVC. Needs -n.
  • Cluster-scoped — belongs to the whole cluster. Node, Namespace, PersistentVolume, StorageClass, ClusterRole. -n is ignored.
  • kubectl api-resources lists every type and its scope.

Three Services can all be called redis. Names must be unique within a namespace, not across the cluster.

  • A namespace is a name scope — a folder for object names.
  • It is the unit for RBAC (who can do what) and ResourceQuota (how much a team may consume).

07 Labels & Selectors

Labels — the glue of Kubernetes

They look decorative in a manifest. They are actually the wiring — every controller in the cluster finds its Pods this way.

pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: vote-abc123
  labels:
    app: vote            # what it is
    tier: frontend       # where it sits
    env: dev             # which environment
    version: v1          # which release
ℹ
Keys may carry an optional prefix: app.kubernetes.io/name. Values: max 63 characters, alphanumeric plus - _ ., and they must start and end alphanumeric.
  • Labels are key/value tags in metadata.labels.
  • They are identifying information — meant to be queried.
  • Kubernetes never interprets them. You define the meaning; controllers only match on them.
  • Nothing in Kubernetes finds a Pod by its name. Deployments, Services, ReplicaSets and NetworkPolicies all find Pods by label.
?
Why labels instead of names — Loose coupling. A Service does not know or care which Pods exist — it asks “everything with app=vote” and gets today’s answer. Pods can be replaced a hundred times and the wiring never changes.

A selector is a query over labels. Two dialects: equality-based and set-based — and modern objects use matchLabels.

Failure state 1

CrashLoopBackOff

The most common failure in Kubernetes, and the one students misread most: the container is starting successfully — and then exiting.

STATUSCrashLoopBackOff
kubectl get pods
NAME     READY   STATUS             RESTARTS      AGE
worker   0/1     CrashLoopBackOff   5 (46s ago)   4m12s
Your container starts fine and then dies. Kubernetes restarts it, it dies again, and the wait between attempts doubles: 10s, 20s, 40s… capped at 5 minutes. The image is fine — the process is not.
⚠
It is not an error in itself — it is Kubernetes telling you it has given up restarting quickly. The real error is in the logs of the run that already ended, which is why --previous matters.
Why it happens
  • The app throws on startup — missing env var, bad config, unreachable dependency. Most common.
  • A dependency is not up yet: worker starting before db exists.
  • The command exits immediately — a container with nothing to do in the foreground.
  • Wrong command/args, or the binary is not at that path.
  • OOMKilled — exit 137, memory limit too low (Day 4).
how to read it
kubectl describe pod worker      # Last State + Exit Code
kubectl logs worker --previous   # logs of the run that DIED
kubectl get events --sort-by=.lastTimestamp | tail

09 Container config

Environment variables in a Pod

The same idea as Compose’s environment: — just a list instead of a map. This is how vote learns what to put on the ballot.

vote-pod.yaml
spec:
  containers:
    - name: vote
      image: iti/vote:v1
      env:
        - name: OPTION_A          # literal value
          value: "Tabs"
        - name: OPTION_B
          value: "Spaces"
        - name: MY_POD_IP         # from the Pod itself
          valueFrom:
            fieldRef:
              fieldPath: status.podIP

          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName
  • env is a list, not a map — every entry needs name plus one of value / valueFrom.
  • value must be a string. Quote your numbers or the api-server rejects the manifest.
  • valueFrom.fieldRef is the Downward API — the Pod reading its own metadata: metadata.name, metadata.namespace, status.podIP, spec.nodeName.
  • Later this week: configMapKeyRef and secretKeyRef pull values from real config objects (Day 3).
⚠

Overriding the image

command vs args

Two optional fields that replace what the image was built to run. Set them rarely — but know exactly what they do.

override both
spec:
  containers:
    - name: demo
      image: busybox:1.36
      command: ["sh", "-c"]              # replaces ENTRYPOINT
      args: ["echo hi; sleep 3600"]      # replaces CMD
override only the arguments
spec:
  containers:
    - name: redis
      image: redis:alpine
      args: ["--maxmemory", "100mb"]     # image ENTRYPOINT kept
  • 1
    Neither set — the image’s ENTRYPOINT and CMD run. This is what you want almost every time.
  • 2
    args only — ENTRYPOINT kept, CMD replaced. The usual way to pass flags to an image.
  • 3
    command only — ENTRYPOINT replaced and CMD is dropped entirely. This one catches people out.
  • 4
    Both set — the image’s own settings are ignored completely. Total control.
⚠
No shell unless you ask — Both are YAML lists, not shell strings. command: ["echo hi && sleep 60"] does not work — there is no shell to interpret &&. If you need shell syntax, ask for one: ["sh", "-c", "…"].

Why the short name works at all

Search domains and ndots

The magic is ordinary resolver config — the kubelet writes it into every Pod.

Beyond one container

Multi-container Pods — what is actually shared

Containers in a Pod share an IP and can share volumes — they do not share a filesystem.

One Pod, two containers
app
helper
shared network namespaceone Pod IP · one port space
shared volumeswhatever you mount into both
NOT shared: filesystems, processes
  • They reach each other on localhost. Same network namespace, so the helper curls http://localhost:80 — no Service, no DNS, no IP.
  • One port space. Two containers cannot both bind :80 in one Pod — the second crashes with address already in use.
  • Root filesystems stay separate. To share files you must mount the same volume (usually an emptyDir) into both.
  • Scheduled and killed together, always on one node. If you would ever want to scale them independently, they are two Pods.
⚠
The most common misuse — Do not put your app and its database in one Pod. The test is simple: would you ever scale, upgrade or restart them separately? If yes — and for a database it is always yes — they are separate Pods.

Ordering, at the platform level

Init containers — run first, run to completion

A container that must finish successfully before your app is allowed to start.

worker waits for db
spec:
  initContainers:
    - name: wait-for-db
      image: postgres:14
      command:
        - sh
        - -c
        - until pg_isready -h db -p 5432; do
            echo waiting for db; sleep 2; done
  containers:
    - name: worker
      image: iti/worker:v1
      imagePullPolicy: IfNotPresent
  • They run in order, one at a time, and each must exit 0 before the next starts. Only when all of them succeed do the app containers start.
  • A failing init container is retried per the Pod’s restartPolicy — the Pod sits in Init:0/1 or Init:CrashLoopBackOff, and the app never starts. That is the desired behaviour.
  • They can use a different image from the app — put psql, git or curl in the init container and keep the app image clean.
  • Classic uses: wait for a dependency, run a schema migration, fetch config or a secret into a shared emptyDir, fix volume permissions.
☞
kubectl logs <pod> -c wait-for-db — init container logs need the -c flag, and a Pod stuck at Init:0/1 is telling you its dependency is not there.

The pattern you will meet in production

The sidecar pattern

A long-running helper container that adds a capability the app does not have to know about.

📝

Log shipper

App writes to a file in a shared emptyDir; the sidecar tails it and ships it out. The app never learns your logging stack exists.

🛡

Service mesh proxy

Envoy in every Pod, intercepting all traffic for mTLS, retries and tracing. This is what Istio and Linkerd actually are.

🔄

Config reloader

Watches a mounted ConfigMap for changes and signals the app to reload — no restart, no coupling.

  • A sidecar runs alongside the app for the whole life of the Pod — long-running, not run-to-completion.
  • The value is separation of concerns: your app keeps doing one thing, the platform concern lives in its own container with its own image and release cycle.
  • It is the second-most-used Pod pattern after “just one container”, and the reason multi-container Pods exist at all.
ℹ
The modern spelling — Since Kubernetes 1.29 there is a proper sidecar: an init container with restartPolicy: Always. It starts before the app containers, keeps running alongside them, and shuts down after them — which finally fixes log shippers dying before the app finishes writing.
⚠
A sidecar shares the Pod’s CPU and memory budget and its lifecycle. Three sidecars around a small app is three times the resource footprint and three more things that can crash the Pod.

Make worker wait for db

cat <<'EOF' | kubectl apply -f -

kind: Deployment
metadata: {name: worker, namespace: vote}
spec:
  replicas: 1
  selector: {matchLabels: {app: worker}}

    metadata: {labels: {app: worker}}

        - name: wait-for-db
          image: postgres:14
          command: ['sh', '-c',

      containers:
        - name: worker
          image: iti/worker:v1
          imagePullPolicy: IfNotPresent
EOF
  • postgres:14 is reused as the init image purely because it already ships pg_isready — and the node already has the image, so there is nothing to pull.

  • The app container keeps its own imagePullPolicy: IfNotPresent. Easy to drop that line while you are concentrating on the init container.
?
Why bother if the app retries? — The Java worker already retries in a loop, so it does not crash — it just sits there logging Waiting for db while kubectl get pods cheerfully reports 1/1 Running. ConfigMaps — take config out of the image

Anything that changes between environments belongs outside the image.

📦

One image, many environments

The same vote image runs in dev, staging and prod. Only the configuration differs — so it must not live inside the image.

🔑

Key/value, plain text

A ConfigMap is a namespaced object holding string key/value pairs. Values can be a word, a number, or a whole file.

🔄

Change config, not code

Edit the ConfigMap and roll the Deployment. No rebuild, no new tag, no docker push.

🗑

Decoupled lifecycle

The ConfigMap outlives the Pods that read it. Delete every Pod — the config is still there for the replacements.

📋

Two ways in

Injected as environment variables, or projected as files in a mounted volume. Same object, two delivery paths.

⚠

Not for secrets

ConfigMap data is stored and displayed in the clear. Passwords, tokens and keys belong in a Secret — next section.

The big misconception

base64 is an <i>encoding</i>, not encryption

Everyone assumes the blob is protected. It is a reversible text transform with no key at all.

prove it in five seconds
kubectl get secret db-secret -n vote -o yaml
# data:
#   POSTGRES_PASSWORD: UzNjdXJlLUlUSS0yMDI2

echo 'UzNjdXJlLUlUSS0yMDI2' | base64 -d
# S3cure-ITI-2026

# no key. no password. no permission beyond `get`.
echo -n 'S3cure-ITI-2026' | base64
# UzNjdXJlLUlUSS0yMDI2   <- round trip
❌

What base64 is not

It has no key, no algorithm choice and no secret. It exists so binary data survives a text field — nothing more.

📁

So: never commit Secret YAML

A base64 blob in git is a plaintext password in git. Use Sealed Secrets, SOPS, or an external store (Vault, AWS Secrets Manager).

🔑

What actually protects it

RBAC on get secrets, encryption at rest in etcd, and not putting the value on a laptop in the first place.

23 · Health

Three probes, three different failures

All three are optional. All three are how Kubernetes finds out whether your app is actually working.

💉

Liveness — is it alive?

Fixes: a process that is running but wedged — deadlocked thread pool, infinite loop, stuck event loop.
On failure: the kubelet kills and restarts the container.

🟢

Readiness — can it serve?

Fixes: traffic arriving before the app can answer — warmup, migrations, a lost downstream dependency.
On failure: the Pod is removed from Service endpoints. No restart.

⌛

Startup — has it booted yet?

Fixes: a slow starter being murdered by its own liveness probe — JVM warmup, big migrations.
On failure: restarts, but liveness and readiness are suspended until it first succeeds.

⚠
Confusion #4 — Liveness restarts. Readiness removes from traffic. Say it back to yourself before writing any probe. Getting these two backwards is the single most common probe bug in production, and it turns a brief blip into a cluster-wide restart storm.
?
Without probes, Kubernetes only knows whether the process is running. It has no idea whether the application works. Probes are how you teach the platform what healthy means for your app.

The probe timeline

startup gates liveness · readiness gates traffic · liveness gates the container’s life.

  • 1
    t=0 · container starts. With no probes at all the Pod is marked Ready right now and traffic arrives immediately — whether or not anything is listening.
  • 2
    Startup probe runs. Liveness and readiness are held back. It may fail up to failureThreshold times (× periodSeconds) — that is your entire boot budget.
  • 3
    Startup succeeds → it never runs again. Liveness and readiness now both begin.
  • 4
    Readiness gates traffic, continuously. Pass → the Pod is in the Service’s endpoint list. Fail → it is pulled out, and put back the moment it passes again.
  • 5
    Liveness watches for wedging, continuously. failureThreshold consecutive failures → the container is killed and restarted, and RESTARTS ticks up.

Liveness or readiness? A decision you make once per symptom

Failure mode

The classic mistake: an over-aggressive liveness probe

A probe that is stricter than the application can ever satisfy makes a healthy app permanently unhealthy.

what a restart loop looks like
livenessProbe:
  httpGet: {path: /healthz, port: 80}
  initialDelaySeconds: 1     # app needs 8s to boot
  periodSeconds: 2
  timeoutSeconds: 1          # slow under load
  failureThreshold: 1        # zero tolerance
the symptom
kubectl get pod -l app=vote -n vote
# NAME         READY   STATUS             RESTARTS
# vote-x2k9d   0/1     CrashLoopBackOff   7 (20s ago)

    The container starts. The app needs 8 seconds; the probe begins after 1.
  • 2
    Three probes fail before the app has bound its port.
  • 3
    failureThreshold: 1 is reached instantly — the kubelet kills the container.
  • 4
    It restarts… and the timer resets. The app is never given long enough to become healthy.
  • 5
    CrashLoopBackOff, forever. The application was fine all along — the probe was the outage.
⚠
A bad liveness probe is a self-inflicted outage. Look at the Killing events in describe: if they say “failed liveness probe”, suspect the probe before the app.
☞
The fix is almost always the same three moves: add a startupProbe to cover boot, raise timeoutSeconds to something humane, and raise failureThreshold to 3. Then ask whether you needed liveness at all.

Interview questions — Day 3

Eight questions that come up in almost every Kubernetes interview.

1

ConfigMap vs Secret

Q: What is the actual difference?
A: Mechanically almost none — both are key/value, both inject as env or files. Secrets are base64 in the API, hidden from describe, held in tmpfs, and gated by their own RBAC verb.

2

Are Secrets encrypted?

Q: Is a Secret encrypted?
A: No. base64 is encoding, not encryption. Encryption at rest in etcd is an opt-in EncryptionConfiguration the cluster admin enables.

3

Config hot reload

Q: I changed a ConfigMap — why did nothing happen?
A: Env vars are fixed at container start; mounted files refresh in about a minute. Either way run kubectl rollout restart unless the app re-reads the file.

4

emptyDir lifetime

Q: When is an emptyDir deleted?
A: When the Pod is removed from the node — not when a container restarts. It is scratch space, never storage.

5

ReadWriteOnce

Q: Can two Pods mount an RWO volume?
A: Yes — if they are on the same node. RWO restricts nodes, not Pods. Use ReadWriteOncePod for true exclusivity.

6

PVC stuck Pending

Q: My PVC is Pending. Why?
A: describe pvc and read the events: no matching StorageClass, no PV that satisfies the size or accessMode — or it is WaitForFirstConsumer and simply has no Pod yet.

7

Liveness vs readiness

Q: Difference between the two?
A: Liveness failure restarts the container; readiness failure removes the Pod from Service endpoints with no restart. Never point liveness at a dependency.

8

Rollouts and readiness

Q: Does a rolling update guarantee zero downtime?
A: Not on its own. Without readiness a Pod joins the Service the moment the process starts. Readiness plus maxUnavailable: 0 is what makes it true.

Two numbers, two different jobs

Requests schedule. Limits constrain.

The single most common cause of a broken production cluster is a container that declared neither.

deployment.yaml · every container
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 256Mi
  • requests — what the scheduler reserves. It is a promise the node makes, subtracted from allocatable capacity whether you use it or not.
  • limits — what the kernel enforces at runtime. Nothing to do with placement; purely a ceiling.
  • No requests? The scheduler assumes zero and will happily cram your Pod onto a node that has nothing left to give.
  • No limits? One runaway container can starve every neighbour on the node.
⚠
Say it out loud — “Requests schedule, limits constrain.” If a Pod is Pending, look at requests. If a Pod is being killed or is slow, look at limits. They are never the same conversation.

What actually happens when you cross the line

At the limit: throttled or killed

Same idea, two completely different outcomes — and this is the exam question.

CPU over limit
compressible · survivable
container wants 800m
↓
limit is 500m
↓
kernel THROTTLES it

The container keeps running, just slower. The CFS scheduler simply hands it fewer time slices. Symptom: latency, not a crash. Nothing in kubectl get pods changes — you have to look at metrics to even notice.

Memory over limit
incompressible · fatal
container wants 300Mi
↓
limit is 256Mi
↓
kernel OOM-KILLS it

There is no such thing as running with less RAM. The kernel OOM killer terminates the process with SIGKILL. The container exits 137, the Pod reports OOMKilled, and the kubelet restarts it — straight into CrashLoopBackOff if it happens again.

Node runs out
pressure · eviction
node memory low
↓
kubelet raises pressure
↓

Different mechanism entirely. Nobody exceeded a limit — the node is short. The kubelet evicts whole Pods to reclaim, choosing victims by QoS class. Status shows Evicted, not OOMKilled.

A class you are assigned, not one you pick

QoS classes

Three classes, decided entirely by which of the four numbers you filled in.

  • 1
    Does every container set both requests and limits, with requests == limits for CPU and memory? → Guaranteed
  • 2
    Otherwise, does at least one container set a request or a limit for anything? → Burstable
  • 3
    Nothing set anywhere in the Pod? → BestEffort
  • You never choose a QoS class. Kubernetes derives it from what you wrote, and stamps it on the Pod at admission time.
  • Read it back with kubectl get pod X -o jsonpath='{.status.qosClass}'.
  • Guaranteed is the only class that gets exclusive CPU pinning on a node with the static CPU manager — and the last to be evicted.
  • BestEffort is free real estate for the scheduler and the first thing thrown overboard.
⚠
A Pod is only Guaranteed if every container qualifies. One forgotten sidecar with no limits drops the entire Pod to Burstable — and people spend hours wondering why.

Eviction order

Who gets thrown overboard first

When a node runs short, QoS class decides the casualty list.

OrderClassWhy it goes
1stBestEffortAsked for nothing, promised nothing. Zero cost to reclaim.
2ndBurstable over its requestUsing more than it reserved. Ranked worst-offender first.
3rdBurstable under its requestStill inside its promise — only touched under real pressure.
lastGuaranteedNode made a hard promise. Evicted only to save the kubelet itself.
  • Eviction is a node-level decision under memory or disk pressure — not a reaction to any one Pod misbehaving.
  • An evicted Pod is deleted from that node, then rescheduled by its Deployment — a bare Pod is simply gone.
  • kubectl get pod shows Evicted; describe node shows the pressure condition that caused it.
☞
Practical rule for this course: give everything a request, give memory a limit, and be careful with CPU limits — an aggressive CPU limit throttles you into slowness that looks exactly like a bug.

Pinning, the simple way

nodeSelector — the blunt instrument

One map of labels on the Pod, matched against the labels on the node. That is the whole feature.

label the node, then demand it
kubectl label node iti-worker disktype=ssd
kubectl get nodes -L disktype
pod spec
spec:
  nodeSelector:
    disktype: ssd
  • Exact key/value match against node labels. Every label must match — it is an AND, never an OR.
  • No match anywhere in the cluster → the Pod stays Pending indefinitely. It does not fall back.
  • Nodes ship with useful labels already: kubernetes.io/hostname, kubernetes.io/arch, kubernetes.io/os, node-role.kubernetes.io/control-plane.
  • Three lines of YAML and no expressiveness at all — which is exactly why node affinity exists.

Soft rules, set operators

Node affinity — the expressive one

Everything nodeSelector does, plus “prefer, but do not insist”.

required vs preferred
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: disktype
          operator: In
          values: ["ssd","nvme"]
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 80
      preference:
        matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["eu-west-1a"]
  • required… is a hard filter — same power as nodeSelector, but with operators: In, NotIn, Exists, DoesNotExist, Gt, Lt.
  • preferred… is a scoring hint with a weight of 1–100. If nothing matches, the Pod still schedules — just somewhere less ideal.
  • That soft mode is the thing nodeSelector can never do, and the reason affinity exists.
  • IgnoredDuringExecution means: once bound, the Pod stays put even if the node stops matching.
☞
The names are long but they parse: required-during-scheduling, ignored-during-execution. Kubernetes is telling you exactly when the rule applies and when it stops caring.

The lock and the key

Taints & tolerations

Taints are the only mechanism where the node gets a say in what runs on it.

🔒 Taint — on the node
repel everything
kubectl taint node iti-worker2 \
  tier=data:NoSchedule
  • “Keep out, unless you carry the key.”
  • The node repels every Pod by default.
  • Remove one by re-running the command with a trailing -.
🔑 Toleration — on the Pod
carry the key
tolerations:
- key: tier
  operator: Equal
  value: data
  effect: NoSchedule
  • “I am allowed past this particular lock.”
  • A toleration permits, it never attracts.
  • operator: Exists tolerates any value of that key.
⚠

Scaling behaviour

The algorithm, and why it is calm

Up quickly, down slowly, and never on a metric it cannot compute.

The formula
one line of maths
desired = ceil( current * ( currentMetric / targetMetric ) )
  • 4 Pods at 80%, target 50% → ceil(4 × 1.6) = 7 Pods.
  • currentMetric is the average across all ready Pods, expressed as a percentage of the CPU request.
  • Inside a 10% tolerance of the target, it does nothing at all.
  • Scale up is fast — it may act on every 15s tick, and can double the replica count in one go.
  • Scale down is slow — a 5-minute stabilisation window by default. It waits for the highest recommendation of the last 5 minutes before shrinking.
  • That asymmetry is deliberate: over-provisioning costs money, under-provisioning costs your users. Flapping costs both.
  • Tune it under spec.behavior (scaleUp / scaleDown policies) if you must — the defaults are good.
⚠
The #1 HPA failure — An HPA without CPU requests does nothing — silently. Utilisation is a percentage of the request, so with no request there is no denominator. The HPA reports TARGETS: <unknown>/50%, never scales, and emits no error you would notice. Requests are not optional here; they are the input.

Workloads that end

Job — run it once, then stop

Every controller so far kept Pods alive forever. This one wants them to finish.

job.yaml
apiVersion: batch/v1
kind: Job
metadata: {name: db-init}
spec:
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: psql
        image: postgres:14
        command: ["sh","-c"]
        args: ["psql -h db -U postgres -f /sql/schema.sql"]
  • A Job creates Pods and keeps creating them until the required number exit 0. Then it stops — permanently.
  • The Pod is not deleted when it finishes. That is on purpose: its logs are the record of the run. Clean up with ttlSecondsAfterFinished.
  • restartPolicy must be OnFailure or Never. Always is rejected — a Job that always restarts would never complete.
  • Status you care about: COMPLETIONS 1/1 and kubectl logs job/db-init.
ℹ
Real uses: database migrations, schema seeding, a one-off batch import, a backup, a data-fix script. Anything you would have run once over SSH now belongs in a Job — declarative, logged, and repeatable.

Picking the right controller

Job or Deployment?

Same Pod spec, opposite definitions of success.

↻ Deployment
  • The process should run forever.
  • Exit 0 is a failure — restart it.
  • restartPolicy: Always, and nothing else is legal.
  • Scale means more concurrent capacity.
  • Success = READY 3/3 and staying that way.
  • vote, result, worker, redis, db.
✔ Job / CronJob
  • The process should run to completion.
  • Exit 0 is the goal — stop.
  • restartPolicy: OnFailure or Never.
  • Scale means more work items finished.
  • Success = COMPLETIONS 1/1, then silence.
  • Migrations, backups, imports, nightly reports.
☞
The test is one question: should this process ever exit? If yes, it is a Job. Running a migration as a Deployment is a classic beginner mistake — it succeeds, exits, gets restarted, succeeds again, and lands in CrashLoopBackOff while having worked perfectly every single time.

Writing the rules

Host rules, path rules and TLS

Two ways to split traffic, and the block that turns on HTTPS.

one Ingress, both styles
spec:
  ingressClassName: nginx
  rules:
  - host: vote.localtest.me
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: vote
            port:
              number: 80
      - path: /result
        pathType: Prefix
        backend:
          service:
            name: result
            port:
              number: 80
  tls:
  - hosts: ["vote.localtest.me"]
    secretName: vote-tls
  • Host-based — routes on the Host header. vote.example.com and result.example.com hitting one IP. Needs real DNS in production.
  • Path-based — routes on the URL path. / to vote, /result to result. No DNS changes needed at all.
  • pathType: Prefix is what you want. Exact matches the whole path, ImplementationSpecific hands the decision to the controller.
  • Longest matching path wins — /result beats /, so ordering in the file does not matter.
ℹ
TLS — TLS termination — awareness only. Put a Secret of type kubernetes.io/tls (a cert and a key) in the tls: block and the controller serves HTTPS on :443 and talks plain HTTP to your Pods.   • 1
    The kubelet is told to start a Pod. It creates the Pod’s network namespace — empty, no interfaces, no address.
  • 2
    It calls the CNI plugin on that node, handing it the namespace and the Pod’s identity.
  • 3
    The plugin allocates an IP from the node’s slice of the pod CIDR, creates a veth pair, and plugs one end into the Pod.
  • 4
    The plugin programs routes so that IP is reachable from every other node — via overlay tunnelling or plain routing.
ℹ
Kubernetes itself contains no networking implementation. It defines a contract and delegates. That is why a fresh cluster with no CNI has all its nodes stuck NotReady — nothing has claimed the job.
The three rules of the Kubernetes network model
  • Every Pod gets its own routable IP. Not a port on the host — a real address of its own.
  • Every Pod can reach every other Pod directly, on any node, without NAT. The source IP the receiver sees is the sender’s actual Pod IP.
  • Agents on a node (kubelet, system daemons) can reach all Pods on that node.
☞
This is why Kubernetes networking feels simpler than Docker’s. No port mapping, no -p 8080:80, no port collisions — two Pods can both listen on :80 because they own different IPs. The flat model is the whole trick.
ℹ
Common plugins: Calico (policy-first, BGP or VXLAN), Flannel (simple overlay, no policy), Cilium (eBPF), kindnet (kind’s tiny default, no policy), AWS VPC CNI (real VPC IPs).

The default nobody expects

NetworkPolicy: default allow, until it is not

An empty cluster is an open network. Policies do not filter — they opt Pods into filtering.

🔓 By default: everything talks to everything
  • A brand-new cluster is completely flat. Any Pod can open a connection to any other Pod, in any namespace, on any port.
  • Nothing is filtered. Nothing is logged. There is no implicit boundary anywhere.
  • Your vote frontend can talk straight to db on 5432. So can a compromised sidecar. So can anything an attacker manages to run.
🔒 A policy flips one Pod to deny-by-default
  • A NetworkPolicy selects Pods with a podSelector. Only those Pods are affected.
  • The moment a Pod is selected by any policy, it switches to deny by default for the directions that policy names.
  • From then on, only traffic explicitly allowed by some policy gets through. Multiple policies are additive — they union, they never subtract.
  • There is no deny rule in the API. You deny by not allowing.
⚠
Namespaces are not network isolation. A Pod in dev can reach a Pod in prod by its FQDN with nothing stopping it. Namespaces separate names, RBAC and quotas — never packets. NetworkPolicy is the only thing in Kubernetes that separates packets.

Start closed, open deliberately

The default-deny pattern

Nine lines of YAML that turn a namespace from open to allow-list.

deny all ingress in this namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
spec:
  podSelector: {}
  policyTypes:
  - Ingress
☞
podSelector: {} means every Pod in this namespace. No ingress: block means no traffic is allowed in. Nine lines, and the namespace is sealed.
  • 1
    Deny first. Apply default-deny to the namespace. Everything breaks — on purpose, and while you are watching.
  • 2
    Allow deliberately. Add one policy per legitimate path, each naming its source and its port.
  • 3
    Verify. Confirm the app works and that a probe Pod is still refused.
  • This is allow-listing: the security model that fails closed. Adding rules one by one to an open network fails open, and you will never know which hole you missed.
  • Scoped to a namespace, because that is what podSelector: {} covers. There is no cluster-wide NetworkPolicy in core Kubernetes.
  • Add - Egress to policyTypes to seal outbound too — then remember to allow DNS to CoreDNS on UDP 53 or nothing will resolve.

Writing real rules

Ingress, egress and the three selectors

Two directions, three ways to name a peer, and one indentation trap.

the shape of a rule
spec:
  podSelector:
    matchLabels:
      app: db
  policyTypes: [Ingress, Egress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: worker
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: batch
    - ipBlock:
        cidr: 10.0.0.0/8
        except: [10.0.5.0/24]
    ports:
    - protocol: TCP
      port: 5432
  egress:
  - to:
    - namespaceSelector: {}
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - {protocol: UDP, port: 53}
  • ingress = who may connect to the selected Pods. egress = where the selected Pods may connect out to. Direction is always from the selected Pod’s point of view.
  • podSelector — Pods by label, in the same namespace as the policy.
  • namespaceSelector — every Pod in namespaces matching those labels. Every namespace carries kubernetes.io/metadata.name automatically.
  • ipBlock — a raw CIDR, for things outside the cluster. Has an except: list.
⚠
OR vs AND — Watch the YAML dashes. Separate list items are OR: - podSelector then - namespaceSelector means either. Two keys under one dash are AND: - namespaceSelector… plus podSelector… means that Pod in that namespace. One character of indentation changes the meaning completely.

The problem Helm solves

Copy-pasting the same YAML per environment stops scaling at about the second environment.

Without Helm
  • k8s/dev/, k8s/staging/, k8s/prod/ — three near-identical copies of the same twelve manifests.
  • The image tag differs. The replica count differs. The hostname differs. Everything else is duplicated.
  • A change to the Deployment shape means editing it three times, and forgetting the third.

  • Public charts exist for almost everything: Postgres, Prometheus, ingress-nginx, cert-manager.

DaemonSet — one Pod per node

Deployment scales by demand. DaemonSet scales by node count.

daemonset.yaml
apiVersion: apps/v1

ResourceQuota and LimitRange

Requests and limits, enforced from above instead of hoped for.

ResourceQuota — a ceiling for the namespace
quota.yaml
kind: ResourceQuota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    pods: "20"
    persistentvolumeclaims: "5"
  • Caps the total across every Pod in the namespace.
  • Exceed it and the Pod is rejected at creation — a clear error, not a Pending Pod.

LimitRange — defaults and bounds per container
limitrange.yaml
kind: LimitRange
spec:

      cpu: 500m
      memory: 256Mi
    defaultRequest:
      cpu: 100m
      memory: 128Mi
    max:
      cpu: "2"
  • Injects defaults into containers that declared nothing — no more BestEffort Pods by accident.
  • Enforces min and max per container.

A method, not a hunch

The six-step playbook

Same six steps, every time, in this order. Discipline beats intuition here.

  • 1
    Look at the object. kubectl get pod -o wide — STATUS, READY, RESTARTS, AGE and NODE. Four of those five are clues.
  • 2
    Read the Events. kubectl describe pod POD and go straight to the bottom. Scheduling, pulls, mounts and probes all report here.
  • 3
    Read the app’s own logs. kubectl logs POD, and --previous if it has restarted — the interesting output belongs to the container that died.
  • 4
    Get inside. kubectl exec -it POD -- sh. Check env vars, mounted files, and whether the app answers on localhost.
  • 5
    Test the network hop by hop. port-forward to the Pod, then to the Service. If the Pod works and the Service does not, it is the selector or the endpoints.
  • 6
    Compare to what works. kubectl get pod GOOD -o yaml > a.yaml, same for the broken one, then diff. The answer is in the diff.

Reading Events properly

Events are the cluster narrating its own decisions. Most people never read them.

events, four ways
# the bottom of describe -- start here
kubectl describe pod vote-7d9f-x2k4

# every event in the namespace, newest last
kubectl get events --sort-by=.lastTimestamp

# only the bad news
kubectl get events --field-selector \
  type=Warning --sort-by=.lastTimestamp

# everything about one object
kubectl get events --field-selector \
  involvedObject.name=vote-7d9f-x2k4

# follow them live during a rollout
kubectl get events -w
  • Events are written by controllers, not by your app: the scheduler, the kubelet, the Deployment controller. Each one is a component telling you what it tried and what happened.
  • They expire after one hour by default. An empty event list on an old failure means nothing — not that nothing went wrong.
  • Read the REASON column first: FailedScheduling, Failed, BackOff, Unhealthy, Killing, FailedMount.
  • kubectl get events is not sorted by default. Always pass --sort-by=.lastTimestamp or you will read them in arbitrary order and reach the wrong conclusion.
☞
Events answer “what did Kubernetes do?”. Logs answer “what did my app do?”. Almost every wasted debugging hour comes from asking one of those questions of the wrong source.

Best practices

Production readiness checklist

Twelve lines. If you can tick them all, the thing is genuinely production-ready.

Before it ships
  • Pinned image tags. vote:1.4.2, never :latest — and ideally a digest.
  • Requests and limits on every container. Memory limit always; CPU request always.
  • All three probes considered. Readiness nearly always, liveness carefully, startup for slow boots.
  • State on a PVC, never in the container filesystem or on hostPath.
  • Secrets externalised — out of the image, out of git, injected at runtime.
  • More than one replica for anything stateless, so a node can die.
Around it
  • Its own namespace, with a ResourceQuota and a LimitRange.
  • NetworkPolicy on the data tier at minimum — nothing reaches the database that does not need to.
  • HPA wherever load varies, backed by real requests.
  • Every manifest in git, applied from CI, never from a laptop.
  • --record or a CHANGE-CAUSE annotation on rollouts, so rollout history is readable.
  • A rollback you have actually rehearsed, not one you assume works.

Run this list against the Voting App you just built. Be honest about which lines you would fail today — that gap is your first week’s work in any real job.

Learn these from other people

The top ten mistakes

Every one of these is common, every one is cheap to avoid, and every one has caused a real outage.

⚠

1 · No resource requests

The scheduler assumes zero, packs the node, and everything degrades together. Also silently disables your HPA.

🏷

2 · :latest tags

Two Pods of the same Deployment can run different code. Rollback becomes meaningless because there is nothing to roll back to.

🔑

3 · Secrets in git

base64 is not encryption. Once it is in history it is public forever — rotate the credential, do not just delete the commit.

🔄

4 · No readiness probe

Traffic hits Pods that cannot serve yet. Every rolling update drops requests — the zero-downtime myth from Day 2.

💣

5 · Aggressive liveness probe

A slow app under load fails the probe, gets restarted, comes back slower. A liveness probe can turn a hiccup into an outage.

📦

6 · Bare Pods in production

Nothing recreates them. The node reboots and your app is simply gone. Always a Deployment, StatefulSet, Job or DaemonSet.

📁

7 · Everything in default

No quotas, no boundaries, no way to delete one app cleanly, and RBAC that cannot be scoped to anything useful.

📜

8 · No CHANGE-CAUSE

rollout history full of <none>. You know a revision 4 exists; you have no idea what it changed.

💾

9 · hostPath for app data

Ties the Pod to one node’s disk. Reschedule and the data is on the wrong machine. Use a PVC.

🔒

10 · No NetworkPolicy

Any compromised Pod anywhere reaches your database directly. Default-allow is a choice you are making by not choosing.

The cheap 80%

Security basics

Five settings that cost nothing and remove most of the easy attacks.

a sane securityContext
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 10001
    fsGroup: 10001
  containers:
  - name: vote
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    resources:
      limits:
        memory: 256Mi
  • Do not run as root. A container escape as UID 0 is a very different afternoon from one as UID 10001. Set runAsNonRoot and let the cluster refuse the Pod if the image ignores it.
  • Drop all capabilities, then add back only what you truly need. Almost no web app needs any.
  • Read-only root filesystem where you can — mount an emptyDir for scratch space instead.
  • No secrets in the image and none in git. Inject at runtime; rotate anything that ever leaked.
  • Scan your images (trivy image vote:1.4.2) and keep base images small — less installed means less to exploit.
ℹ
Clusters increasingly enforce this centrally with Pod Security Admission — label a namespace pod-security.kubernetes.io/enforce=restricted and non-compliant Pods are rejected at creation. Better to write compliant manifests now than to discover it at deploy time.

What it costs to run

Resource and cost hygiene

The gap between requested and used is real money, on someone’s real invoice.

  • You pay for requests, not for usage. A cluster 95% requested and 10% used costs the same as one that is genuinely busy. This is where cloud bills go to die.
  • Size requests from measurements. Run under real load, read kubectl top, set the request just above the observed steady state — then revisit it.
  • Memory limits: yes. CPU limits: think. A memory limit protects the node from a leak. An aggressive CPU limit just throttles your app into latency that looks like a bug.
  • Scale to what you need. An HPA with a sane minReplicas beats a fixed count guessed at peak.
  • Delete what you are not using. Finished Jobs, evicted Pods, orphaned PVCs and unattached load balancers all keep costing.

Eight you will genuinely be asked

Day 4 interview questions

Answer each in two sentences. Interviewers are testing the model, not your vocabulary.

  • 1
    What is the difference between a resource request and a limit? Requests are used by the scheduler to place the Pod and are reserved on the node; limits are enforced at runtime by the kernel. Requests schedule, limits constrain.
  • 2
    What happens when a container exceeds its CPU limit? Its memory limit? CPU is throttled — it just runs slower. Memory is not compressible, so the container is OOMKilled with exit code 137 and restarted.
  • 3
    Name the three QoS classes and how a Pod gets each. Guaranteed: every container sets requests == limits for CPU and memory. Burstable: something is set but they do not match. BestEffort: nothing set. Eviction goes BestEffort first, Guaranteed last.
  • 4
    Difference between nodeSelector, node affinity, and taints? nodeSelector and affinity are Pod-side attraction — affinity adds operators and soft preferences. Taints are node-side repulsion; a toleration only grants permission, it does not attract.
  • 1
    Why might an HPA never scale? No metrics-server, or — far more often — no CPU request on the container. Utilisation is a percentage of the request, so with no request there is no metric. It shows <unknown> and fails silently.
  • 2
    Job versus Deployment? A Deployment keeps a process running forever and treats exit 0 as a failure. A Job runs to completion and treats exit 0 as success. Jobs must use restartPolicy OnFailure or Never.
  • 3
    Do namespaces isolate network traffic? No. Any Pod can reach any Pod in any namespace by default. Only a NetworkPolicy — enforced by a CNI that supports it — restricts traffic.
  • 4
    How does a NetworkPolicy actually work? It selects Pods with a podSelector; those Pods flip to deny-by-default for the directions listed, and only explicitly allowed traffic passes. Policies are additive and there is no deny rule.
